# 前端页面设计
能够展示音标、译文、长难句分析、点击句子时展示该句子的核心词汇、词组、语法结构。

两种模式：
1. 阅读模式：展示原文、音标、译文、循环播放音频。
2. 学习模式：展示原文、音标、长难句分析（点击句子时展示该句子的核心词汇、词组、语法结构）。

音标展示：
上面是英文，下面是音标。英文和音标对齐。像中文和拼音对齐一样。
json中音标和英文是一整句的形式保存的，因此页面展示时需要通过空格切分单词和音标。

布局：
左右两侧面板，6:4。左侧是原文和音标，右侧是译文/长难句分析。

# 数据处理 (create_audio.py) 建议架构

既然音频文件是通过句子的 MD5 命名，这个脚本可以做到增量生成（也就是不重复生成已经存在的文件，节约时间和接口额度）：

脚本读取 article.json。
遍历所有段落和句子，提取 original_text。
计算该句子的 MD5 值，拼接出路径 data/audio/MD5.mp3。
判断该文件路径是否已存在：如果不存在，则调用 google TTS 语音合成接口，并保存为对应的 MP3。


# 后端接口设计
轻量级前后端分离 如果未来需要管理多篇文章，如何设计接口让前端自动生成页面？ 后端本质上只需扮演数据提供者的角色，不参与页面拼接。

1- 获取文章列表接口: GET /api/articles —— 返回所有文章的 title 和 id 列表。
2- 获取文章详情接口 (驱动页面生成的核心): GET /api/articles/{article_id} —— 后端直接读取对应的 article.json 并作为 payload 返回。
3- 前端拿到 JSON 后：
    * 如果是 阅读模式：只遍历 JSON 渲染左侧的句子+音标，右侧集中渲染全部的 translation，开启“音频 <audio> 播放完自动定位并播放下一句”的逻辑。
    * 如果是 学习模式：给左侧的每个句子绑定一个 onClick 事件，点击时，将当前句子的 analysis 树结构提取出来，用组件渲染到右侧 40% 的面板中展示长难句和词汇字典。
